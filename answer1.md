## 一、怎么在“很忙”的情况下积累（关键：极简化）

别做复杂知识库，做这个就够了：

### ✅ 方法：**一页卡片法（One Page Playbook）**

每次只记录一件事，用 1 分钟写完：

```
【场景】
客户忙 / 需求不清 / 工期紧 / 出现分歧...

【我怎么做】
一句话描述动作（比如：先干+发声明）

【关键动作】
- 提案（写依据）
- 明确假设
- 发声明（默认方案）
- 留证据（邮件/Teams）

【结果】
推进了 / 被否 / 被调整

【可复用原则】
以后遇到类似情况：可以直接这么干
```

👉 本质：
不是记录“事情”，而是**抽象“动作模板”**

---

### ✅ 触发机制（关键，不然你不会做）

你不可能每天整理，但可以**只在3种时刻记**：

1. **刚踩坑 / 刚吵完 / 刚被怼**
2. **一个事推进成功**
3. **你感觉“这次做得挺对”**

👉 每周3条就够，半年就是资产库

---

### ✅ 存储方式（不要折腾工具）

直接用最简单的：

* 飞书 / Notion / Excel / 甚至微信收藏
* 一个目录就行：

```
01 推进类
02 沟通类
03 报告类
04 风险控制
05 客户博弈
```

## 二、把它变成“随时能用”的能力（比记录更重要）

### 🔹推进类

1. **客户忙 = 默认推进机制**

   * 提案 + 写依据 + 发声明 + 默认执行
   * 客户有异议再调整

2. **不要等审批，先给“可被否决的方案”**

   * 没方案 → 卡死
   * 有方案 → 可推进

---

### 🔹沟通类

3. **所有口头结论 → 必须文字化**

   * 目的：避免认知偏差 + 责任留痕

4. **模糊问题 → 强制具体化**

   * 谁负责 / 截止日 / 下一步动作

---

### 🔹报告类

5. **报告只写3件事**

   * 客户关心什么
   * 现在卡在哪
   * 下一步谁干什么

6. **“球在谁手里”必须写清**

   * 不写 = 默认在你手里

---

### 🔹风险控制

7. **所有妥协必须留依据**

   * 为什么这么做
   * 有什么风险
   * 谁认可的

---

### 🔹客户博弈

8. **推进优先级 > 完美合规**

   * 先活下来，再优化

9. **用“声明”替代“等待确认”**

   * 等确认 = 风险在你
   * 发声明 = 风险共担
